# Теория — M1-W02

## 1. Зачем своя обёртка (gateway) над LLM
Прямой вызов провайдера из клиента/фронта плох: утекает ключ, нет единой точки логирования, стоимости, ретраев, смены модели. Архитектурный паттерн — **собственный gateway**:
- ключ живёт только на сервере;
- одна точка для логов, метрик, стоимости (пригодится в M4);
- можно сменить провайдера, не трогая клиентов;
- можно добавить кэш, лимиты, фильтры.

Это первое «архитектурное» решение недели и кандидат в ADR.

## 2. FastAPI за 5 минут (что важно)
- Эндпоинт — функция с декоратором (`@app.post("/ask")`).
- Тело запроса/ответа описывается моделями Pydantic — это и валидация, и контракт, и авто-документация (OpenAPI/Swagger по адресу `/docs`).
- Асинхронность (`async def`) полезна, т.к. вызов LLM — это ожидание сети.

## 3. Контракт через Pydantic
Модели = граница доверия. Невалидный вход отсекается до бизнес-логики.
```python
class AskRequest(BaseModel):
    question: str
    temperature: float = 0.0

class AskResponse(BaseModel):
    answer: str
    model: str
    tokens: int
```
Контракт сразу даёт тебе `openapi.yaml` — артефакт, который оценит любой системный аналитик/архитектор.

## 4. /health
Маленький эндпоинт, возвращающий `{"status": "ok"}`. Нужен, чтобы Docker/оркестратор/мониторинг понимали, жив ли сервис. Без него прод-обвязка не умеет проверять сервис.

## 5. Ошибки и таймауты
LLM-провайдер может тормозить или падать. Минимум на этой неделе:
- таймаут на запрос к провайдеру;
- понятная ошибка клиенту (502/503), а не stack trace;
- (на M4 добавим ретраи и fallback).

## 6. Почему тесты мокают LLM
Тест не должен ходить в платный API: это медленно, дорого и нестабильно (ответ каждый раз разный). Мокаем функцию вызова LLM и проверяем **свою** логику: валидацию, формат ответа, обработку ошибок. Это и есть тестируемая архитектура.
