# Теория — M1-W01

Цель: дать ровно то понимание LLM, которое нужно архитектору решений, чтобы принимать технические решения и ставить задачи. Без вывода формул внимания.

## 1. Что такое LLM (по-инженерному)
LLM предсказывает следующий токен по предыдущему контексту. Для архитектора важны не слои трансформера, а следствия этой природы:
- ответ **вероятностный**, а не вычисленный из базы фактов;
- модель **не знает**, где правда, а где правдоподобная выдумка;
- качество ответа сильно зависит от качества контекста, который вы подали.

Практический вывод: LLM — это не «база знаний», а «генератор текста по контексту». Всё проектирование (RAG, источники, проверки) — это способы дать модели правильный контекст и поймать её ошибки.

## 2. Токены
Токен — минимальная единица текста для модели (часть слова, слово, символ). Русский текст обычно токенизируется хуже английского — больше токенов на тот же смысл, значит дороже и быстрее упираешься в лимит.

Следствия:
- длинные документы не помещаются в один запрос -> нужен RAG (разбиение + поиск);
- стоимость считается в токенах (вход + выход) -> FinOps;
- надо управлять контекстом, а не «кидать всё подряд».

## 3. Context window
Это максимум токенов, который модель учитывает за один запрос (промпт + ответ). Превысил — старое «забывается» или запрос отклоняется. Архитектурно это бюджет, который вы распределяете между системным промптом, контекстом (RAG), историей и ответом.

## 4. Temperature и детерминизм
Temperature управляет случайностью выбора следующего токена:
- 0 — почти детерминированно, повторяемо (нужно для извлечения данных, классификации, тестов);
- 0.7–1 — разнообразнее, «креативнее» (нужно для черновиков текста).

Для бизнес-логики и тестируемости по умолчанию берут низкую temperature. Это первое архитектурное решение, которое вы зафиксируете в ADR.

## 5. Галлюцинации
Галлюцинация — правдоподобный, но не подтверждённый ответ. Это не баг, а свойство вероятностной генерации. В корпоративной системе ответ LLM нельзя принимать без обвязки:
- **источники** (provenance) — откуда взято;
- **confidence** — насколько модель/система уверена;
- **human-in-the-loop** — человек подтверждает важные решения;
- **eval-набор** — измеряем долю ошибок;
- **логирование** — можем разобрать инцидент.

Это и есть работа архитектора: не «использовать LLM», а обложить её контролем так, чтобы бизнес мог ей доверять.

## 6. Prompt vs context
- **Prompt** — инструкция (system: «ты ассистент, отвечай только по документам»; user: вопрос).
- **Context** — данные, которые вы вкладываете в запрос (фрагменты документов, история, результаты инструментов).

Большая часть инженерии LLM — это управление контекстом (context engineering), а не подбор «магической фразы».

## 7. Structured output (почему это важно с первой недели)
Свободный текст backend обработать надёжно не может. Нужен JSON по схеме, например:
```json
{
  "answer": "string",
  "confidence": 0.82,
  "sources": ["doc_12#p3"],
  "requires_human_review": true
}
```
JSON можно валидировать (Pydantic), тестировать, логировать, маршрутизировать в workflow. Поэтому structured output — не «продвинутая фича», а база корпоративного LLM-сервиса.
